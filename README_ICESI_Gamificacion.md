# ICESI Interactiva - Gamificación Ingeniería de Software

## Descripción General

Esta actividad consiste en una experiencia gamificada basada en un partido de fútbol, donde los participantes asumen diferentes roles de la ingeniería de software y toman decisiones para avanzar en la cancha hasta anotar goles.

Cada reto representa una fase del desarrollo de software y se basa en la toma de decisiones en contextos realistas.

---

## Estructura General

Flujo base:

Inicio → Contexto → Decisiones → Avance (pases) → Gol

- Decisión correcta: avance directo
- Decisión subóptima: ruta más larga
- Decisión incorrecta: penalización y corrección

---

## RETO 1: Definiendo el producto

### Contexto

Una startup llamada QuickBite quiere desarrollar una app de pedidos de comida. El cliente solicita una solución completa en poco tiempo.

### Flujo

S1: Inicio  
Decisión:
- Empezar a programar
- Definir requerimientos (correcta)
- Diseñar interfaz

Ramas:
- Programar sin requerimientos → confusión → volver a definir requerimientos
- Diseñar primero → desalineación → volver a definir requerimientos

S3: Priorización  
Decisión:
- Hacer todo al tiempo
- Priorizar funcionalidades clave (correcta)
- Empezar por lo más fácil

Resultado: Gol

### Conceptos

- Requerimientos
- MVP
- Comunicación con cliente

---

## RETO 2: Construcción del sistema

### Contexto

El equipo ya conoce lo que debe construir y ahora debe diseñar e implementar el sistema.

### Flujo

S1: Arquitectura  
Decisión:
- Todo en un bloque
- Separar frontend/backend/base de datos (correcta)
- Solo frontend

Ramas:
- Monolito desorganizado → refactorización
- Solo frontend → error conceptual → corrección

S3: Lógica de desarrollo  
Decisión:
- Mostrar menú sin validar
- Validar login antes (correcta)
- Pedir login siempre

Resultado: Gol

### Conceptos

- Arquitectura por capas
- Flujo lógico
- Diseño estructurado

---

## RETO 3: Validación y despliegue

### Contexto

La aplicación está lista y se debe decidir si se lanza o se valida previamente.

### Flujo

S1: Pre-lanzamiento  
Decisión:
- Lanzar directamente
- Realizar pruebas (correcta)
- Revisar visualmente

Ramas:
- Fallos en producción → testing obligatorio
- Validación incompleta → corrección

S3: Manejo de errores  
Decisión:
- Ignorar errores pequeños
- Corregir antes de lanzar (correcta)

S4: Despliegue  
Decisión:
- Proceso automatizado (correcta)
- Despliegue manual

Resultado: Gol

### Conceptos

- Testing
- Calidad
- Despliegue
- Impacto en usuarios

---

## Modelo de progresión

Cada reto permite múltiples rutas:

- Ruta óptima: menos decisiones, avance rápido
- Ruta alternativa: más pasos, mismo resultado
- Ruta incorrecta: penalización y corrección obligatoria

---

## Objetivo educativo

El participante comprende:

- Cómo se toman decisiones en ingeniería de software
- La importancia de los roles dentro del equipo
- El flujo real de desarrollo de software
- El impacto de decisiones técnicas y de negocio

---

Resultado final: Software funcional (Gol)
